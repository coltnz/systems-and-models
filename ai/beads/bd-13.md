## Problem (reviewer, [P1], r3337754930)
`assembleExtraction` still trusts model-supplied `d.id` for atoms and relationships. The OpenAI
structured-output schema only constrains these to strings, so **duplicate** atom ids or duplicate
relationship ids are valid model output → validator reports `duplicate_id` (graph error) →
`postPackDraft` persists it (only blocks structural) → the review UI can't edit ids or delete a
duplicate → permanent 409-on-reviewed-save — the same dead-end class bd-12 fixed for dangling
endpoints. This is the last model-controllable graph error that can strand a pack (the other graph
checks are server-set fields or already throw/drop at assembly).

## Fix — dedupe by id at the shared assembly chokepoint (keep-first)
In `packages/extraction/src/draft.ts` `assembleExtraction`:
1. Dedupe draft atoms by `id`, keeping the FIRST occurrence (deterministic, order-preserving); build
   the atom-id Set from the deduped atoms (so the bd-12 dangling-endpoint check still resolves a
   reference to the kept first atom).
2. Dedupe draft relationships by `id`, keeping the first, ALONGSIDE the bd-12 dangling-endpoint drop +
   empty-`anchor_ids` normalize + `UnknownAnchorError` throw.
3. `output_ids` lists only the kept (deduped) atom + relationship ids (no duplicates).
Strategy: drop-keep-first over rename (renaming a dup id could silently break a relationship endpoint
that referenced it) or whole-draft reject (more user-hostile than dropping one bad item).

## Definition of done / tests (packages/extraction/src/index.test.ts)
- duplicate atom ids → only first kept (in atoms + output_ids); pack validates, no `duplicate_id`.
- duplicate relationship ids → only first kept; pack validates.
- mixed bundle (dup atom + dup rel + bd-12 dangling edge + empty-anchor edge) → only unique/valid/
  normalized items kept; `output_ids` has no duplicates; pack validates clean.
- all existing bd-12 + extraction tests still pass.
- mock path unaffected → committed `examples/*.json` byte-unchanged and still validate.
