# bd-16 — Serialize same-pack mutations (issue #4)

- **Status:** open
- **Type:** implementation (issue #4 follow-up)
- **Depends on:** bd-7
- **Blocks:** —

## Problem (D-015)
The server does read-modify-write on a pack per mutation with no per-pack lock, so two concurrent
mutations on the same pack id each read the same version, modify independently, and the second
`putPack` silently overwrites the first — one edit lost, no error. Low for the single-user alpha (the
UI disables buttons while busy) but a real correctness gap.

## Fix
Add an in-memory per-key serialization (async mutex / promise-chain) in `packages/server/src/index.ts`
keyed by pack id, so the read-modify-write-validate-persist cycle for a given pack runs to completion
before the next mutation on that same pack starts. Apply to all pack-mutating routes (patchAtom,
splitAtom, patchRelationship, postReviewed). Different pack ids still run concurrently. Pure `handle`
core stays correct; the mutex lives at the route-dispatch or per-pack-operation boundary. No persistence
/ DB change. Deterministic; injected now/idFactory unaffected.

## Definition of done / tests
- Two concurrent `PATCH` ops on the SAME pack (e.g. accept atom A and accept atom B fired without
  awaiting in between) both land — neither edit is lost (test via the pure core / `handle` with
  interleaved promises, asserting the final pack reflects BOTH mutations).
- Concurrent ops on DIFFERENT packs are not serialized against each other (no deadlock; test).
- All existing server tests still pass; full root gate green; examples byte-unchanged.

## Constraints
`packages/server/src/index.ts` (+ tests) only. No new deps. In-memory only (single process). Don't
change route shapes or status codes.
