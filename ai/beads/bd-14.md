## Problem
A fresh review pass (server/persistence, validator/tutor/types, input robustness) found one real
bug plus several robustness gaps. Headline: **path traversal via id fields**. The `Store`
(`store.ts`) builds `join(dataDir, sub, \`${id}.json\`)` with NO sanitization, and `source_id`/`pack_id`
come straight from request bodies — so `{"source_id":"../../etc/passwd"}` escapes `dataDir`, and a
tutor `pack_id` like `../packs/<draftId>` defeats the reviewed-only invariant and crashes
`answer(undefined,…)` → 500 that echoes the attempted path. Localhost-only (low blast radius) but it's
exactly the "client-controllable input → unrepairable/crashing sink" class.

## Fixes (each with a test; keep all gates green; examples unchanged)
1. **[security] Sanitize ids at the Store chokepoint.** Add `isSafeId(id)` = matches
   `^[A-Za-z0-9._-]+$` and id ∉ {`.`,`..`}. In `readJson`: an unsafe id → return `undefined` (→ route
   404). In `writeJson`: an unsafe id → throw a clear error (server-minted ids are always safe; guard
   anyway). This closes every traversal vector (getSource/getPack/getReviewed/put*). Our real ids
   (`src-<hex>`, `srv-…`, demo `id-N`) all pass. Tests: `getSource("../x")`/`getPack("../x")`/
   `getReviewed("../packs/x")` return undefined; a body `source_id:"../../etc/passwd"` to
   `/packs/draft` → 404 (not a read outside dataDir, no crash); tutor `pack_id:"../packs/<id>"` →
   refusal (no `answer(undefined)` throw).
2. **[robustness] Tutor route defense-in-depth.** In `postTutorQuery`, if the loaded reviewed snapshot
   is falsy OR lacks a `.pack`, return a refusal (never call `answer` with a non-pack). Test.
3. **[robustness] Reject empty/whitespace-only `content`** at `POST /sources` → 400 (avoids the silent
   zero-anchor empty pack). Test.
4. **[robustness] Cap request body size** in the `node:http` listener (~10 MB); abort + 413 instead of
   buffering unbounded. Test (oversized body → 413, server stays up).
5. **[hygiene] Don't leak internal `e.message`** from the top-level `handle` 500 catch — return a
   generic "internal error" (keep specific 4xx messages for known validation/ingest/bad-id cases, and
   keep logging server-side if desired). Test that a forced internal error returns a generic 500 body.
6. **[ui correctness] `NewSourceForm.loadExisting()`** currently hardcodes `validation:{ok:true,
   errors:[]}` when loading an existing pack via the dropdown — so a stored pack with errors shows
   "valid" and the reviewed-save button isn't blocked. Call `validate(packId)` (POST /packs/:id/validate)
   after `getPack` and use the real result. Test (App/NewSourceForm) that a loaded pack with a
   structural error renders blocking + disables save.
